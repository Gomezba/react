El valor inicial

Introducción

Ahora que ya sabemos que el valor del estado se gestiona en un almacenamiento externo conocido como store, vamos a centrarnos en entender el momento en que ese estado se crea y como React gestiona su valor inicial.

useState sin parámetro

El valor inicial del estado se puede enviar a este useState a través de su primer y único parámetro y puede ser de varios tipos.

Ya sabemos que cuando no se pasa el parámetro ya que es un valor opcional, nos devuelve undefined.

El no pasar el parámetro para que nos devuelva undefined se usa en muchos casos.

useState con un un valor fijo

Pasamos un valor fijo como lo puede ser true que en este caso nos sirve para inicializar, pero puede ser cualquier otro valor fijo como un texto plano.

Como podemos ver, todo nos aparece como activo, ya que estamos inicializando todos los componentes con su estado en true es por eso que todos los usuarios parten del estado activo y evidentemente su botón está marcado como "Desactivar".

Lo mismo pasará si lo hacemos con false solo que empezará con el estado inactivo.

Hasta aquí todo parece sencillo, podemos pasar un simple valor y este useState simplemente lo utilizará para crear el estado.

Cada vez que damos click al botón, el valor del estado cambia y no se vuelve tomar en cuenta el valor inicial, solo se ocupa una vez y es para declarar su valor inicial.

useState con una variable

Como ya hemos visto, el estado puede depender de algún tipo de variable como puden ser las props.

Tenemos la prop que en este caso se llama state y es la prop que determina si el estado está activo o no.

Ahora agregaremos la prop state en consola para ir viendo los valores del estado y de la prop.

Como podemos ver, al principio, es decir sin haber dado click, nos aparecen los valores iniciales de los estados que le hemos dado a los componentes, y a su vez podemos ver el valor de la prop.

Pero si damos click al botón, ya estamos cambiando el estado del componente al que damos click, y en consola podemos ver que el valor del estado ha cambiado, pero el valor de la prop siempre se mantendrá con el valor inicial, siempre será el mismo nunca cambiará.

Como vemos volvemos otra vez al mismo caso de un valor fijo, esta prop state solamente determina el valor inicial de este estado, solamente sirve para cuando el estado se crea por primera vez en la store y a partir de ahí no se toma en cuenta para nada, en ningún caso.

Hasta aquí hemos ido con los casos sencillos, pero es que este useState puede admitir tambien como parámetro una función y aquí es donde se complica un poco toda esta lógica.

useState con una función como parámetro

Imaginemos que queremos que este estado sea aleatorio el estado inicial y queremos calcularlo a través de un Math.random.

Pasando una función de manera incorrecta

Lo que haremos es crear una función llamada randomActive para devolver un booleano y pondremos un console.log para ver cuando se ejecuta.

A continuación estaremos viendo el primer renderizado que nos da esto.

Si recargamos el navegador varias veces iremos viendo que el valor inicial del estado irá cambiando.

Vemos que esto funciona, el valor inicial se está asignando de manera aleatoria.

Pero estamos teniendo un problema, y es que si miramos la consola podremos ver que la función se está ejecutando 3 veces.

Esto es normal ya que se debe ejecutar una vez para cada componente, pero hay que hablar de que pasa cuando cambia el estado, por eso hemos puesto el console.log dentro de la función.

Tenemos que darnos cuenta que pasa cuando hacemos un setActiveState, es decir, cuando damos click a un botón del componente.

Entonces cada vez que hacemos click, se ejecutará la función randomActive(), es decir aunque ya el valor inicial no importe y el valor venga determinado por el estado como tal, este randomActive se sigue ejecutando cada vez que cambiamos un estado.

Obtener el valor inicial de un estado a partir de una función es algo muy común que se puede dar en muchos componentes y lo que no queremos es que esta función se tenga que ejecutar en cada render, una función que evidentemente a partir del primer renderizado, cuando se crea este estado no tiene ningún sentido que se siga ejecutando.

Pasando una función de manera correcta

Para este caso lo que React nos permite pasar a este useState es pasar una función como tal, en lugar de su ejecución directa, es decir, pasamos la función randomActive como callback.

Forma incorrecta ✖️

/* De esta manera lo que haciamos es pasar el resultado
de una función en lugar de pasar una función*/
useState(randomActive()) // ✖️

Forma correcta 👍

//Pasamos randomActive como callback
useState(randomActive)

En el primer renderizado se ejecuta esta función randomActive 3 veces, una para cada componente.

Pero si ahora cambiamos el estado ejecutando el setActiveState, es decir, dando click al botón, miraremos que la función randomActive ya no se ejecutará

Como vemos la función randomActive no se vuelve a ejecutar nunca más, es decir que en este caso si que estamos logrando nuestro objetivo, estamos consiguiendo que esta función solamente se ejecute cuando es realmente necesario cuando queremos obtener de verdad el valor inicial.

En este punto nos podemos preguntar, ¿Pero este randomActive se ejecuta siempre?, es decir, ¿No hay ninguna forma de evitarlo?, pues la realidad es que no.

¿Y que implica esto?, pues que no podemos almacenar funciones en el estado. A partir del siguiente setActiveState si podrimos setear el estado con una función, pero realmente aquí ya nos estamos dando cuenta de que no deberia haber una función como estado.

El estado tiene que almacenar el mínimo valor posible para representar la información que almacena, en el caso del activeState lo que necesitamos es un valor booleano, un true o un false, no tiene ningún sentido que almacenemos cosas como, el texto, como la clase, o cualquier otro valor que podamos cálcular a partir de ese estado.

Como por ejemplo lo que habiamos hecho anteriormente, cuando calculamos un valor en base a un booleano, esto es correcto.

{activeState ? 'Desactivar' : 'Activar'}

No tendría sentido almacenar en el estado el texto como tal si podemos calcularlo a partir de un booleano.

En el estado siempre hay que almacenar la mínima información posible.

Desde luego que no tiene ningún sentido que aquí intentasemos almacenar alguna función.

Lo que tendríamos que hacer es intentar almacenar un valor que nos permita obtener una función u otra dependiendo de el.

Si nuestra función necesita un parámetro, necesitamos envolver la función dentro dentro de otra, de la siguiente manera.

// Si nuestra función necesita parámetros la envolvemos dentro de otra función
useState(() => randomActive(param))

Este patrón es totalmente recomendable en situaciones donde necesitas calcular el valor inicial del estado una sola vez y el cálculo depende de una función que no deseas ejecutar en cada renderizado, es por eso que la envolvemos dentro de una función para que no se ejecute en en cada renderizado sino solo una vez.

Esta sintaxis de envolver funciones dentro de otra se ocupa demasiado en el tema de los eventos.

Con esto ya concluimos las 3 casuísticas de que podemos tener para este valor inicial.

  1. useState(): Sin parámetro nos devulve undefined.

  2. useState(valorFijo): Podemos pasarle un valor que puede ser fijo o puede provenir de una variable, de una prop o de cualquier sitio, realmente no es relevante de donde venga sino del valor que tenga.

  3. useState(función): Podemos pasar una función, en el caso de que la pasemos, solamente se ejecuta para obtener este valor inicial y no se ejecuta nunca más

    Si la función necesita un parámetro se necesita envolver dentro de otra función. Esta manera se ocupa mucho en los eventos.

Conclusiones

El estado en React está diseñado para representar datos que afectan la interfaz de usuario y pueden cambiar con el tiempo. El estado no debe usarse para almacenar lógica o comportamientos, como funciones. La lógica de tu aplicación, incluyendo las funciones que manipulan el estado o gestionan eventos, debe definirse fuera del estado.

Al almacenar funciones en el estado, puedes generar complejidad innecesaria y hacer que tu código sea más difícil de entender y mantener. Además, el estado debe ser lo más sencillo y directo posible, conteniendo solo el mínimo de información necesaria para representar la UI.

Ejemplo de almacenamiento incorrecto de funciones en el estado:

 import { useState } from "react";

function MyComponent() {
 // Ejemplo de función almacenada incorrectamente en el estado
 const [myFunction, setMyFunction] = useState(() => () => console.log("Función en el estado"));

 return (
   <div>
     <button onClick={myFunction}>Ejecutar función</button>
   </div>
 );
}

Problema: Aquí, el estado está almacenando una función que imprime en la consola, lo cual no es recomendable porque el estado debería contener datos (valores que describen la UI) y no comportamientos o lógica.

Ejemplo de manera correcta de usar una función para calcular un valor inicial sin almacenar la lógica: 

 import { useState } from "react";

function MyComponent() {
 // Usamos una función para calcular el valor inicial, pero almacenamos el resultado, no la función
 const [active, setActive] = useState(() => randomActive(5));

 return (
   <div>
     <p>Estado inicial calculado: {active ? "Activo" : "Inactivo"}</p>
     <button onClick={() => setActive(!active)}>Cambiar estado</button>
   </div>
 );
}

function randomActive(param) {
 return param > 0 ? true : false;
}

Correcto: Aquí usamos una función (randomActive(5)) para calcular el valor inicial del estado, pero solo almacenamos el resultado de esa función en el estado, no la función en sí. Esto es completamente válido y recomendable, ya que la función se ejecuta solo una vez al inicializar el estado.

Resumen:

  • El estado es para almacenar datos, no lógica ni comportamientos.

  • No almacenes funciones en el estado, ya que eso va en contra de la idea de mantener el estado simple y directo.

  • Si necesitas calcular el valor inicial de un estado, usa una función que devuelva un valor y almacena ese valor, no la función.
