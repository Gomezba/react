CSS modules

Introducción

Hasta ahora hemos modularizado nuestros componentes, prestando únicamente atención a la parte de JavaScript, es decir a la parte de JSX.

Pero un componente no es solo JavaScript, sino que tambien tiene su CSS asociado y actualmente estamos tratando el CSS de forma global, es decir, el mismo CSS para todos.

Hasta aquí todo es muy sencillo, pero a medida que esto va creciendo, el archivo index.css puede convertirse en un auténtico monstruo.

Lo que haremos este este capítulo es modularizar y divir en distintos archivos nuestro index.css

¿Dónde se coloca el CSS de cada componente?

Habitualmente se colocan a lado del propio componente, es decir, en la carpeta de components se crea un archivo de CSS con el mismo nombre del componente y se coloca ahí sus estilos correspondientes.

Para añadir este CSS se hace importandolo desde el mismo componente.

De tal forma que cuando importemos el componente, el componente importará su propio CSS, y solo se cargará este CSS si el componente está importado.

Procedemos a crear los archivos de CSS con el mismo nombre del componente, en nuestro caso los únicos componentes que tienen estilos son: UserRow y UsersList. A su vez agregaremos su respectivo CSS.

UserRow.css

UsersList.css

En el archivo index.css solo se quedan los estilos globales.

Importamos el CSS desde el respectivo componente.

UserRow.jsx

UsersList.jsx

Hasta este punto tenemos nuestro CSS modularizado, y todo funciona correctamente.

Pero aquí no acaba todo, porque aunque este estilo esté importado dentro de un componente, no está restringido únicamente a ese componente. El CSS no se ha marcado con un ámbito específico que lo limite al componente en cuestión.

Problema de colisión de CSS

Como se mencionó anteriormente, el CSS de nuestros componentes a pesar de que los estemos importando desde su respectivo componente, no está ligado a el, no hemos restringido que ese CSS sea único del componente.

El problema que genera esto es que nuestro CSS colisionará

Ejemplo:

Vamos a ir al archivo UsersList.css y cambiaremos la clase por wrapper.

Lo mismo haremos con el componente UsersList.jsx.

Lo mismo haremos con el componente UserRow.css y UserRow.jsx cambiaremos la clase a wrapper .

Si vemos la pantalla, será un completo caos.

¿Qué es lo que está pasando con el proyecto?, pues que nuestras clases wrapper están colisionando, esto se debe a que no son únicas del componente como habiamos mencionado anteriormente, a pesar de que el CSS se esté importando desde su respectivo componente, ese CSS no es único del componente, es por eso que nos colisionan las clases.

Entonces una vez que importamos el CSS, no es único del componente, sino que forman parte de cualquier CSS global.

Para que lo veamos más claro, miremos el código html de nuestros componentes en el navegador.

¿Como podemos solucionar esto y convertir este CSS en algo realmente propio del componente?, pues utilizando CSS modules.

Solucionando colisiones con CSS modules

CSS modules es una forma de crear hojas de estilo para que sean propias de un componente, pero nos obliga a utilizar el CSS de una forma especial.

Convirtiendo CSS en un módulo

El estándar que tenemos en React y en Vite, es que nuestros archivos CSS tengan como nombre:

archivo.module.css

Sabiendo esto, procederemos a cambiar el nombre de nuestros archivos CSS siguiendo esta regla.

Nombre correcto de archivos CSS modules

Importando CSS module de manera incorrecta

Actualizamos los imports del CSS de nuestros componentes a este nuevo nombre.

Si vemos nuestra pantalla miraremos que esto aún no funciona, no está generando nuestros estilos.

No está funcionando porque la manera correcta de importar un CSS module es importandolo como un objeto.

Importando CSS module de manera correcta

Para poder importar de manera correcta nuestro CSS module, necesitamos importarlo como objeto:

import style from 'archivo.module.css'

El nombre del objeto por convención se suele llamar style pero podemos darle el nombre que nos apetezca.

¿Qué es este style?

Es un objeto que contiene todas las clases CSS puestas en forma de objeto.

Miremos un ejemplo:

// archivo.module.css

.clase {
color: tomato;   
}

Para poner nuestras clases en nuestros componentes, necesitamos acceder a ellas a través del objeto style.

<p className={style.clase}> texto </p>

Como style es un objeto no podemos tener propiedades que contengan guiones medios (-), así que si tenemos clases que contengan guiones medios, tendremos que acceder a esas clases con la sintaxis de corchetes y con comillas.

// archivo.module.css

/* Clase en kebab-case */
.clase-example {
color: tomato;   
}    
<p className={style['clase-example']}> texto </p>

En realidad trabajar con la sintaxis de corchetes es muy incomodo, en lugar de nombrar nuestras clases con la nomenclatura kebab-case podemos utilizar la nomenclatura camelCase.

// archivo.module.css

/* Nuestras clases estarán en camelCase */          
.claseExample {
color: tomato;   
}

De esta manera al asignar nuestras clases a nuestras etiquetas JSX, simplemente accedemos a ellas con la nomenclatura del punto. 

<p className={style.claseExample}> texto </p>

Sabiendo todo esto, ahora si nuestras clases en nuestros componentes quedarían de la siguiente manera.

Ahora si que todo funciona correctamente.

¿Que pasa por detrás con CSS modules?

Ya vimos que todo funciona correctamente, pero ¿Qué es lo que está pasando?, pues que estos CSS modules están siendo procesados por el empaquetador.

Si miramos las clases en el HTML del navegador miraremos algo distinto a lo que hemos escrito.

Lo que vemos es que a nuestras clases se le agrega una serie de caracteres generados, para diferenciarlo de la otra clase wrapper.

Es decir que lo que viene dentro de este className={style.wrapper} no es simplemente la clase como tal, sino un nombre autogenerado que le da el empaquetador a esta clase.

Conclusiones

Puede ser que no nos parezca la mejor forma de tratar clases y puede ser que nos resulte incomodo, pero la realidad es que esta es una de las mejores formas de garantizar que el CSS tiene el ámbito que debe tener, en este caso para nuestros componentes.

Obviamente en proyectos reales, podemos utilizar arquitecturas de CSS, existen frameworks de CSS como tailwind que nos proporciona utility clases.

Existen mil alternativas para una arquitectura de CSS y está es solo una más de todas ellas, y está es la más habitual que solemos encontrar tanto en los cursos de React como en muchos de los proyectos, porque nos porporciona esta ventaja, pero no quiere decir que sea la única.
