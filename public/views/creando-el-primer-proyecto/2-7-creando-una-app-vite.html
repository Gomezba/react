Creando una app con Vite

¿Por qué utilizamos Vite?

La alternativa que utilizaremos para crear nuestras aplicaciones de React es vite.

Vite es un framework, que nos permite crear un proyecto no solamente en React, si no, en muchas otras aplicaciones de JavaScript.

Vite, es una alternativa muy popularizada ya que es bastantemente rápido y tiene una amplia gama de características, que hacen que sea idónea tanto para pequeños proyectos para cuando estamos aprendiendo, como para proyectos enormes que tengan que ser llevados a producción con un rendimiento clave.

Creando proyecto con Vite

Empezamos por abrir la terminal en la raiz donde crearemos nuestro proyecto, en nuestro caso será la carpeta react-vite, es decir dentro de esta carpeta se creará un proyecto que tendrá su propia carpeta.

Ejecutamos el comando:

npm init vite@latest

Establecemos un nombre de proyecto, el framework y su variante.

Nos movemos a la carpeta del proyecto que nos a creado Vite, en nuestro caso la llamamos first-project

Ejectamos el comando npm i para instalar las dependencias.

De primer vistazo, podemos ver que, es un proyecto mucho más simple que el que habiamos creado con create-react-app.

A demás, el peso de la carpeta de las dependencias (node_modules) es considerablemente bajo, con solo 94mb a comparación de los 149mb de las dependencias de create-react-app.

Hechemos un vistazo al package.json , observaremos que solo tenemos 4 scripts.

  1. dev: Ejecuta el comando vite para iniciar el servidor de desarrollo. Esto te permite correr tu proyecto en un entorno local con recarga en caliente (hot-reloading), lo que significa que cualquier cambio que hagas en tu código se reflejará instantáneamente en el navegador sin necesidad de recargar la página completa. Esto agiliza el desarrollo, ya que solo se actualizan las partes modificadas del proyecto, manteniendo el estado de la aplicación.

  2. build: Ejecuta vite build para generar una versión optimizada de tu proyecto lista para producción. El resultado es un conjunto de archivos estáticos que pueden ser desplegados en un servidor web.

  3. lint: Ejecuta eslint . para analizar el código en busca de errores o problemas de estilo según las reglas configuradas en ESLint. El punto (.) indica que se revisará todo el proyecto.

  4. preview: Utiliza vite preview para lanzar una versión previa de tu aplicación construida. Esto simula cómo se verá la app en producción al correr un servidor de solo lectura para revisar el resultado final.

Vite en modo desarrollo

Comenzaremos por utilizar el primero de los comandos

npm run dev

y seleccionamos nuestro servidor local.

Y en menos de 1 segundo nuestra app estará ejecutandose

¿Por qué es tan rápido?

Porque en Vite no bandelizamos en desarrollo .

Si nos vamos a la pestaña Sources miraremos que tenemos varios archivos js, lo que quiere decir que solo tenemos una versión transpilada sin llegar a bandelizar.

Como hemos mencionado, cuando estamos en modo desarrollo no es necesario bandelizar la aplicación.

Miraremos esto a detalle en la documentación de Vite: ¿Por qué Vite?

Vite en modo producción

Ejecutamos el comando

npm run build

Esto nos genera rápidamente una carpeta llamada dist con una carpeta dentro llamada assets con los siguientes archivos.

Esta carpeta "dist" es la que subiremos a producción.

Ahora, miraremos el index.html.

El archivo index-DfQ3Eo-l.js, se encuentra bandelizado con todo el código de React y React-dom como todos los demás módulos de nuestra aplicación. A su vez el código se encuentra ofuscado y minificado.

Los estilos de nuestra aplicación se encuentran de igual manera bandelizados (Todos los archivos css se unifican), y a su vez minificados.

¿Y para qué sirve la carpeta public?

Si la carpeta, "dist" es la que subimos a producción, ¿Para qué nos sirve la carpeta public?

En Vite, la carpeta public se utiliza para almacenar archivos estáticos que no pasarán por el proceso de build o transformación de Vite. Estos archivos se copian directamente a la carpeta de salida (dist) tal como están. Los archivos que pongas en public estarán disponibles de manera pública cuando subas tu proyecto a producción.

Por ejemplo, si tienes un archivo logo.png en public, este se podrá acceder en producción como https://tu-dominio.com/logo.png. Es útil para elementos que no necesitas procesar o modificar con JavaScript o CSS, como imágenes, fuentes o manifestos (como favicon.ico o manifest.json).

En resumen, cualquier archivo que pongas en public será copiado directamente a dist sin modificaciones y estará disponible en la raíz del proyecto.

Vite en modo preview

Para asegurarnos de que nuestra aplicación funciona, y antes de subir nuestra aplicación al servidor de producción, Vite, nos proporciona el comando:

npm run preview

para previsualizar nuestra aplicación y ver que todo este correcto y funcione.

Ejecutamos el comando y nos generará un servidor para visualizar.

Abrimos el servidor y visualizaremos nuestra aplicación.

Y ahora si vamos a la pestaña sources miraremos todo bandelizado.

Entonces con Vite estas serían las 2 opciones, en desarrollo no bandelizamos y en producción si.

Configuración

Solo queda hablar de la configuración de Vite.

Vite nos proporciona un archivo llamado vite.config.js, donde podremos configurar todo lo que queramos.

Vite funciona a través de plugins, es decir si tenemos el plugin de React (react()), que nos sirve para configurar un proyecto genérico de Vite como un proyecto de React. Dentro de los paréntesis podremos aplicar un monton de configuraciones de React que extiende la configuración por defecto. Es decir si modificamos una pequeña opción, no tenemos que tomar todo el archivo de configuración del empaquetador y montarlo por nuestra cuenta como se hace con Create React App, si no que simplemente pisamos esa opción y seguimos manteniendo todo tal y como está.

De la misma forma tenemos un monton de plugins oficiales de Vite que nos permiten configurar todo el empaquetador y transpilador.

Y en este caso ¿Cúal sería ese empaquetador? y ¿Cúal sería ese transpilador? pues bien, aquí no estamos utilizando ni Webpack ni Babel, si no que Vite utiliza una combinación entre esbuild y rollup.js. Son 2 herramientas que a priori hacen lo mismo.

¿Por qué utilizar 2 en lugar de solo una?, La razón es muy sencilla. Vite utiliza como base esbuild y todo lo que puede hacer con esbuild lo hace con el. Esbuild es transpilador y tambien es empaquetador, entonces la mayoría de tareas de este empaquetado y esta transpilación al completo, las hace con esbuild .

Pero hay ciertas funcionalidades que esbuild todavía no permite, esbuild es una herramienta que es relativamente nueva y todavía no tiene toda la compatibilidad necesaria, a pesar de que es excesivamente rápida hay pequeñas cosas que aún no puede hacer. Entonces todos esos pequeños fallos que tiene esbuild los corrigen utilizando rollup.js.

Pero evidentemente el plan de Vite es que cuando esbuild evolucione, dejarán de utilizar rollup.js y se quedarán solo con esbuild.
